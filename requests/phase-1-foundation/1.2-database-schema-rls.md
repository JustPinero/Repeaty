# Request 1.2 — Database Schema + RLS Policies

## Objective
Land the full Phase-1 database schema as SQL migrations: `profiles`, `user_languages`, `decks`, `cards`, `reviews`, `pronunciation_attempts`, `comprehension_attempts`. RLS enabled on every user-owned table with explicit policies. The `on_auth_user_created` trigger creates a `profiles` row on signup. Validated by integration tests against a live local Supabase.

## Acceptance Criteria → Test Mapping

| Criterion                                                                          | Test                                                                                                              |
| ---------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| All seven tables exist with the columns/types specified in `references/schema.md`  | `supabase/tests/schema.test.ts` — for each table, query `information_schema.columns` and assert shape             |
| RLS is enabled on every user-owned table                                           | `schema.test.ts` queries `pg_class.relrowsecurity` and asserts `true` for each                                    |
| User A cannot read User B's `profiles`/`user_languages`/`reviews`/attempts rows    | `supabase/tests/rls-isolation.test.ts` signs in as A, inserts; signs in as B, reads — expects empty result        |
| Bundled decks are visible to any authenticated user                                | `rls-isolation.test.ts` — seed a bundled deck via service role, both A and B can `SELECT` it                      |
| AI-generated decks are visible only to their owner                                 | Same test — A creates an `ai_generated` deck, B cannot read it                                                    |
| `auth.users` insert triggers a matching `profiles` row                             | `supabase/tests/triggers.test.ts` — signup via auth; expect `profiles` row to exist with same `id`                |
| `tier` and `is_admin` cannot be self-mutated by a regular user via UPDATE          | `rls-isolation.test.ts` — User A `UPDATE profiles SET tier='pro'` returns 0 rows affected                         |
| Indexes exist on the documented hot paths                                          | `schema.test.ts` queries `pg_indexes` for `idx_reviews_user_due`, `idx_cards_deck_id`, `idx_pron_user_card_created` |
| Soft-delete pattern: `decks.deleted_at IS NOT NULL` rows are not returned by `SELECT *` policies | `rls-isolation.test.ts`                                                                                |

## RED Phase (write these tests FIRST)

Set up `supabase/tests/` running against a fresh `supabase start` instance per CI run:
- `schema.test.ts` — schema shape assertions per the matrix in `references/schema.md`.
- `rls-isolation.test.ts` — multi-user isolation tests using two service-role-created auth users + their JWTs.
- `triggers.test.ts` — auth signup → profile row.

These tests must fail before any migration is written.

## GREEN Phase (implement until tests pass)

Migrations (append-only, in order):
- `0001_init_profiles.sql` — `profiles` table + the `on_auth_user_created` trigger + the `tier` CHECK constraint.
- `0002_user_languages.sql` — `user_languages` table.
- `0003_decks_cards.sql` — `decks` and `cards` tables, foreign keys, denormalized `cards.language_code`, indexes.
- `0004_reviews.sql` — `reviews` table with `UNIQUE (user_id, card_id)` and `idx_reviews_user_due`.
- `0005_attempts.sql` — `pronunciation_attempts` and `comprehension_attempts` with their indexes.
- `0006_rls_policies.sql` — every RLS policy per `references/schema.md`.

After implementation, run `/drift-audit` and confirm `schema.md` matches the migrations exactly.
