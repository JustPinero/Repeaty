# Fix — `ts-fsrs` version spec drift between docs and package.json

## What's wrong
- `packages/shared/package.json:16` declares `"ts-fsrs": "^4.5.0"`.
- `references/architecture.md`'s pre-DriftAudit row said `^4.7.1` (the actual lockfile-resolved version).
- `pnpm-lock.yaml` resolves to `ts-fsrs@4.7.1` for both today.

This DriftAudit run patched the doc to read `^4.5.0` (matching package.json) so the docs aren't lying. But the team should pick a single source of truth and stick with it.

## Why it matters
- The version range `^4.5.0` permits a silent upgrade to any 4.x → 4.999.999 on a fresh install. If a `ts-fsrs` 4.8.0 ships a behavioral change (FSRS scheduler tuning), our review intervals shift without a code change.
- Pinning at `^4.7.1` (the version the team actually validated against) is more honest about what we're depending on.
- Either way, the package.json and the doc must agree.

## Proposed fix
Bump the package.json range to match the doc's intent:

```json
"dependencies": {
  "ts-fsrs": "^4.7.1"
}
```

Then regenerate the lockfile:
```bash
pnpm i
```

Re-update `references/architecture.md` to read `^4.7.1` (this DriftAudit changed it to `^4.5.0` to match the package; that change should be reverted alongside the package.json bump).

Run the FSRS test suite to confirm no behavioral change:
```bash
pnpm --filter @repeaty/shared test
```

## Files to touch
- `packages/shared/package.json` — bump `ts-fsrs` range to `^4.7.1`
- `pnpm-lock.yaml` — regenerated by `pnpm i`
- `references/architecture.md` — restore `^4.7.1` in the Request 2.2 dep row

## Acceptance criteria
- `packages/shared/package.json` and `references/architecture.md` agree on the `ts-fsrs` range.
- `bash scripts/validate.sh` passes.
- `packages/shared/src/fsrs.test.ts` continues to pass (no behavioral regression from the spec bump — we expect none, since the lockfile already resolves there).
