# Request 2.1 — Bundled Decks (ES + FR) + Import Pipeline

## Objective
Seed two bundled starter decks (Spanish A1 ~30 cards, French A1 ~30 cards) into the database via a deterministic, idempotent migration. Add a generic JSON-import pipeline (Python or Node script under `scripts/seed/`) that converts a small structured deck definition (YAML or JSON) into a SQL migration body. After this request, every fresh `supabase db reset` produces the same two bundled decks; the dashboard query under 1.5 actually returns rows once a future request links the user to bundled decks.

## GREEN Phase (implement until tests pass)

1. **Define deck source format.** YAML (one file per deck) at `scripts/seed/decks/`:
   ```yaml
   id: starter-es-a1                # stable slug, becomes deterministic UUIDv5
   name: Spanish — Starter (A1)
   language_code: es
   cefr_level: A1
   cards:
     - target: hola
       native: hello
       example_target: ¡Hola, ¿cómo estás?
       example_native: Hi, how are you?
     - ...
   ```

2. **Author the two starter YAML files** with ~30 cards each. Use a curated greetings/numbers/days/colors mix (CEFR-A1 frequency).

3. **Implement `scripts/seed/seed-decks.ts`** (Node, run via `tsx` or `node --import tsx`):
   - Reads every `*.yaml` in `scripts/seed/decks/`
   - Generates a deterministic UUIDv5 from each deck's `id` slug (namespace constant in the script) so migration content is stable across runs
   - Same approach for cards (UUIDv5 from `${deck-id}/${index}`)
   - Outputs `supabase/migrations/0009_seed_bundled_decks.sql` containing INSERTs guarded by `ON CONFLICT (id) DO UPDATE SET ...` so re-applying is idempotent
   - Includes a header comment with the script's git SHA + run timestamp for traceability

4. **Add `pnpm seed:decks`** to root package.json: `tsx scripts/seed/seed-decks.ts`.

5. **Document in CONTRIBUTING.md** (or `scripts/seed/README.md`) how to add a new deck: edit YAML → run `pnpm seed:decks` → commit the regenerated migration.

6. **Generate and commit `0009_seed_bundled_decks.sql`** as the Phase-2.1 deliverable.

## Notes
- UUIDv5 (not v4) so the migration is byte-stable across regenerations. Without it, every `pnpm seed:decks` produces a different SQL diff and the migration becomes useless as a regen artifact.
- `ON CONFLICT (id) DO UPDATE` (not DO NOTHING) so editing a card's translation in YAML and re-running the script propagates the change on next migration. Cards have FK from reviews, but updating `target_text` doesn't break those FKs.
- 30 cards × 2 decks = 60 cards. Small enough to ship in a single migration without performance concern.
