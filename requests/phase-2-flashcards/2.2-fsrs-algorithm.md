# Request 2.2 — FSRS Algorithm in `@repeaty/shared`

## Objective
Implement the FSRS (Free Spaced Repetition Scheduler) algorithm in `packages/shared/src/fsrs.ts` so it can be imported by both the web client (review session UI) and (eventually) Edge Functions (server-side analytics, stats). FSRS is more accurate than SM-2 for interval prediction; ADR-006 already accepted this trade-off.

The implementation must produce:
- A pure `schedule(state, rating, now)` function — no I/O, no mocks needed in tests.
- An opaque `FsrsState` JSONB blob (matches `reviews.fsrs_state` column).
- A 4-button rating contract: Again / Hard / Good / Easy (1 / 2 / 3 / 4).
- An `initialState()` function for new cards (first review).
- A `dueAt(state, now)` derivation so Dashboard's "X cards due" query is straightforward.

## Acceptance Criteria → Test Mapping

| Criterion                                                                                  | Test                                                                       |
| ------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------- |
| `initialState()` returns a state with `due_at = now`, `interval_days = 0`, `ease ≈ 2.5`    | `fsrs.test.ts > initialState`                                              |
| Rating Again on a new card schedules a re-review within 10 minutes (memory-trace short)    | `fsrs.test.ts > schedule again first review`                               |
| Rating Easy on a new card schedules a re-review at least 4 days out                        | `fsrs.test.ts > schedule easy first review`                                |
| Repeated Good ratings produce monotonically increasing intervals                           | `fsrs.test.ts > monotonic good intervals`                                  |
| Rating Again on a mature card (interval > 30d) drops interval to ≤ 1 day                   | `fsrs.test.ts > again resets mature card`                                  |
| `schedule()` is pure — same (state, rating, now) → same output                             | `fsrs.test.ts > deterministic`                                             |
| Returned `due_at` is always strictly after `last_reviewed_at`                              | `fsrs.test.ts > causality`                                                 |
| `dueAt(state, now)` returns `state.due_at` (formal API)                                    | `fsrs.test.ts > dueAt accessor`                                            |
| State serializes losslessly to JSON (round-trip through JSON.stringify/parse equals input) | `fsrs.test.ts > serialization round-trip`                                  |
| TypeScript types are strict: `Rating` is a literal-union, no `any`                         | type-check enforces                                                        |

## GREEN Phase (implement until tests pass)

1. **Vitest infra in `@repeaty/shared`:** new `package.json` deps + `vitest.config.ts` + `test` script.
2. **Algorithm implementation:** port from a known-good open-source FSRS-4.x reference (Anki community FSRS). Pin version + cite source in a comment header.
3. **Default parameter set:** use the FSRS-4 defaults; keep them as a `const DEFAULT_PARAMETERS` so future Pro features can tune per-user.
4. **Document the rating semantics in the file header:**

   - 2 (Hard): correct but with significant effort
   - 3 (Good): correct with normal effort
   - 4 (Easy): trivial recall
