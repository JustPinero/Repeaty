# Request 5.1 — Pro Tier Infrastructure

## Objective
Schema-side groundwork for Pro features: `rate_limits` (per-user daily counters that gate `generate-lesson` + `generate-feedback`), `feedback_cache` (cross-user cache so identical mistake patterns don't burn redundant Claude calls), and `tier_change_log` (an audit trail for `/admin` tier flips so we can answer "who upgraded Ben to Pro on day N"). All three are spec'd in `references/schema.md` § Auxiliary; this request lands them as migration `0015`.

## Acceptance Criteria → Test Mapping

| Criterion                                                                       | Test                                                                       |
| ------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `rate_limits` exists with composite PK `(user_id, bucket, day)`                 | `rate-limits-rls.test.ts` (integration)                                    |
| `rate_limits` is read-own / write-service-role (anon + authenticated DENY writes) | same                                                                     |
| `feedback_cache` exists with `UNIQUE (card_id, error_pattern, native_language_code)` | `feedback-cache-rls.test.ts`                                          |
| `feedback_cache` reads are public (any authenticated user); writes are service-role only | same                                                                |
| `tier_change_log` records `(actor_id, target_id, from_tier, to_tier, reason, created_at)` | `tier-change-log-rls.test.ts`                                       |
| `tier_change_log` reads gated to `is_admin = true`; writes service-role only    | same                                                                       |
| `bump_rate_limit(p_bucket, p_cap_per_day)` RPC SECURITY DEFINER returns the new count or raises `RATE_LIMITED` (exception code from a custom SQLSTATE) | `bump-rate-limit-rpc.test.ts` |

## GREEN Phase

1. **Migration `0015_pro_tier_infra.sql`:**
   - `rate_limits(user_id uuid, bucket text, day date, count int default 0, primary key(user_id, bucket, day))`. RLS: SELECT `auth.uid() = user_id`; no INSERT/UPDATE/DELETE policy (writes via service-role + the bump RPC only).
   - `feedback_cache(id uuid pk, card_id uuid fk cards, error_pattern text, native_language_code text, feedback_text text, created_at timestamptz, UNIQUE(card_id, error_pattern, native_language_code))`. RLS: SELECT `to authenticated using (true)`; no write policy.
   - `tier_change_log(id uuid pk, actor_id uuid fk auth.users, target_id uuid fk auth.users, from_tier text, to_tier text, reason text, created_at timestamptz)`. RLS: SELECT only when caller has `profiles.is_admin = true`; writes service-role only.
   - `bump_rate_limit(p_bucket text, p_cap_per_day int) returns int` SECURITY DEFINER: upserts the row for `(auth.uid(), p_bucket, current_date)` incrementing `count` by 1; raises `errcode = 'P0001'` with message `RATE_LIMITED` if `count > p_cap_per_day`. Grant execute to authenticated.
2. **Test helper RPC `test_force_rate_limit_count(...)`** that lets the integration suite seed counts without ticking the real bump RPC.
