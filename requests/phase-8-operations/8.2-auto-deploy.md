# Request 8.2 — Auto-Deploy on Merge to Main

## Objective
Wire push-to-`main` → Vercel deploy → Supabase migrations push → Edge Functions deploy → smoke. Today every step is manual: `pnpm build`, `vercel deploy --prebuilt --prod`, `supabase db push`, `bash scripts/deploy-supabase.sh --functions-only`. The risk of getting these out of order (deploying frontend before a backing migration) is the largest live operational footgun.

After this lands, the only manual step left is for a deploy that needs a credential rotation or schema-change-with-data-backfill — which is where manual gates belong.

## Acceptance Criteria → Test Mapping

| Criterion                                                                       | Test                                                                       |
| ------------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| `.github/workflows/deploy.yml` exists, runs on push to `main`                   | manual review                                                              |
| Deploy job runs **after** `validate` + `edge-functions` + `supabase-migrations` succeed (uses `needs:` from `ci.yml` via workflow_run) | manual review |
| Order: migrations → edge functions → frontend → smoke                            | manual review                                                              |
| `supabase db push` runs only with **no destructive ops** (the CLI's default — but explicit in the step's run string) | manual review                                  |
| Vercel deploy uses `vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}` (the prebuilt path proven in 7.1) | manual review                          |
| On smoke failure: workflow runs `vercel rollback --token=...` to promote the previous deployment, then exits non-zero | manual review                            |
| Required secrets are documented in `references/env-vars.md` § GitHub Actions   | manual review                                                              |
| First push after merge actually completes the cycle end-to-end                  | live CI run                                                                |

## RED Phase

`scripts/test-deploy-config.test.ts` — Vitest spec that parses `.github/workflows/deploy.yml` and asserts:
- The `on:` trigger is `workflow_run` for `CI` with `branches: [main]` and `types: [completed]`.
- A guard step bails when `github.event.workflow_run.conclusion != 'success'`.
- Step ordering: `db-push` → `functions-deploy` → `vercel-deploy` → `smoke`.
- The smoke step's `if: failure()` companion runs `vercel rollback`.

Fails RED today; passes once 8.2 ships.

## GREEN Phase

1. **`.github/workflows/deploy.yml`:**
   ```yaml
   name: Deploy
   on:
     workflow_run:
       workflows: [CI]
       branches: [main]
       types: [completed]

   env:
     NODE_VERSION: "22"
     VERCEL_PROJECT_REF: prj_... # from .vercel/project.json
     SUPABASE_PROJECT_REF: pvoupsduyymykawlmhpu

   jobs:
     deploy:
       if: github.event.workflow_run.conclusion == 'success'
       runs-on: ubuntu-latest
       timeout-minutes: 15
       steps:
         - uses: actions/checkout@v6
         - uses: supabase/setup-cli@v2.0.0
         - name: Push migrations
           env:
             SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
             SUPABASE_DB_PASSWORD: ${{ secrets.SUPABASE_DB_PASSWORD }}
           run: |
             supabase link --project-ref "$SUPABASE_PROJECT_REF"
             supabase db push --password "$SUPABASE_DB_PASSWORD"

         - name: Deploy Edge Functions
           env:
             SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
           run: bash scripts/deploy-supabase.sh --functions-only

         - uses: pnpm/action-setup@v6
         - uses: actions/setup-node@v6
           with:
             node-version: ${{ env.NODE_VERSION }}
             cache: pnpm
         - name: Install + build
           run: |
             pnpm install --frozen-lockfile
             pnpm --filter @repeaty/web build
         - name: Vercel build artifact
           working-directory: apps/web
           env:
             VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
             VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
             VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
           run: |
             pnpm dlx vercel@latest pull --yes --environment=production --token="$VERCEL_TOKEN"
             pnpm dlx vercel@latest build --prod --token="$VERCEL_TOKEN"
         - name: Vercel deploy (prebuilt)
           id: vercel
           working-directory: apps/web
           env:
             VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
             VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
             VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
           run: |
             URL=$(pnpm dlx vercel@latest deploy --prebuilt --prod --token="$VERCEL_TOKEN" 2>&1 | tail -1)
             echo "url=$URL" >> "$GITHUB_OUTPUT"
         - name: Smoke against new deploy
           run: pnpm smoke
         - name: Rollback on smoke failure
           if: failure() && steps.vercel.outcome == 'success'
           env:
             VERCEL_TOKEN: ${{ secrets.VERCEL_TOKEN }}
             VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
             VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
           working-directory: apps/web
           run: pnpm dlx vercel@latest rollback --token="$VERCEL_TOKEN" --yes
   ```
2. **Required GitHub repo secrets** (set via `gh secret set` or the repo settings UI):
   - `VERCEL_TOKEN` — personal access token from https://vercel.com/account/tokens.
   - `VERCEL_ORG_ID` + `VERCEL_PROJECT_ID` — from `apps/web/.vercel/project.json` (already created during 7.1).
   - `SUPABASE_ACCESS_TOKEN` — personal access token from https://supabase.com/dashboard/account/tokens.
   - `SUPABASE_DB_PASSWORD` — the cloud project's database password.
3. **Document in `references/env-vars.md`:** new "GitHub Actions" section listing the four secrets and where each comes from.
4. **First-time setup runbook in `requests/phase-8-operations/8.2-auto-deploy.md`** itself (this section), including how to test in a throwaway PR before merging.

## Out of scope
- Auto-deploy from feature branches (Vercel preview deploys remain Vercel's automatic behavior — we don't manage those from CI).
- Slack notification on deploy success/failure.
- Database migration **reversal** automation. Migrations are forward-only per `references/deployment-landmines.md`; a bad migration needs a hand-authored follow-up reversal migration.
- Edge Function rollback. `supabase functions deploy <name>` from a prior commit is the manual recovery path; documenting this remains the rollback mechanism.

## Risks
- **Rollback only covers the frontend.** If a migration goes through but the smoke fails on a frontend regression, `vercel rollback` undoes the JS — but the migration is still applied. Mitigation: order is migration-first, frontend-last, and the smoke covers the surface area where a migration breakage would show. The bigger danger is a migration that passes smoke but fails in user code paths — caught only by E2E or live use.
- **Race with manual `vercel --prod`.** While 8.2 is in flight, a `vercel --prod` from local would interleave with CI. Convention after 8.2 ships: never `vercel --prod` by hand — push to main instead.
